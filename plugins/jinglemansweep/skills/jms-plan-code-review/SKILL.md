# jms-plan-code-review

Perform a holistic code review of the project codebase and write structured issues to a YAML review file. This is a standalone, user-invocable skill primarily called by the `jms-plan-execute` orchestrator after all tasks complete and between fix rounds. It focuses on integration correctness, architectural consistency, security, and code quality -- not stylistic preferences.

## Usage

```text
/jms-plan-code-review <phase-path> [scope]
```

**Arguments:**

- `<phase-path>` -- Path to the phase directory (e.g. `.plans/2026/03/01/01-auth-implementation`). This directory must contain `tasks.yaml` and `prd.md`.
- `[scope]` -- (Optional) One of:
  - `full` -- Review the entire codebase. Used on the first review run after all tasks complete.
  - `changed` -- Review only files changed since the last review round. Used during fix rounds.

If `scope` is omitted, default to `full`.

**Directory convention:**

```text
<phase-path>/
├── prompt.md           <- input (original requirements)
├── prd.md              <- input (approved PRD)
├── tasks.yaml          <- input (task list)
├── reviews/            <- output directory
│   ├── round-1.yaml    <- output (generated by this skill)
│   ├── round-2.yaml    <- output (subsequent runs)
│   └── ...
├── fix-ledger.yaml     <- input (if exists, from jms-plan-fix)
├── logs/               <- created by jms-plan-phase-new
└── state.yaml          <- created by jms-plan-phase-new
```

## Instructions

### Step 0: Determine Repository Root

Run `git rev-parse --show-toplevel` to determine the repository root. All `.plans/` references in this skill resolve relative to this root (i.e. `<repo-root>/.plans/`). If the command fails (not a git repository), stop and tell the user this skill requires a git repository.

### Step 1: Validate Inputs

1. Verify that `<phase-path>` exists and contains both `tasks.yaml` and `prd.md`. If either is missing, stop and tell the user which file is missing and which skill generates it (`/jms-plan-prd` for `prd.md`, `/jms-plan-task-breakdown` for `tasks.yaml`).
2. Verify that the `reviews/` subdirectory exists inside the phase directory. If it does not exist, create it.
3. Determine the review round number by counting existing `round-*.yaml` files in the `reviews/` directory and adding 1. For example, if `round-1.yaml` and `round-2.yaml` exist, this run produces `round-3.yaml`.

### Step 2: Read Context

Read the following files to build context for the review:

1. `<phase-path>/prd.md` -- Understand the project requirements, acceptance criteria, and scope.
2. `<phase-path>/tasks.yaml` -- Understand what was implemented, which files were affected, and how tasks relate to requirements.
3. `<phase-path>/fix-ledger.yaml` -- If it exists, read it to understand what issues have already been identified and fixed in prior rounds. Do not re-raise issues that have been resolved.
4. Previous review round files (`reviews/round-*.yaml`) -- If any exist, read them to understand what issues have already been raised. Do not duplicate issues from prior rounds unless they remain unresolved after a fix attempt.

### Step 3: Determine Review Scope

**If scope is `full`:**

Build the list of files to review from the `files_affected` fields across all tasks in `tasks.yaml`. Also include any additional files that were created or modified during implementation but not listed in `tasks.yaml` -- detect these by checking `git diff` against the base branch or by scanning the project for recently modified files.

**If scope is `changed`:**

Determine which files were modified since the last review round. Use one of the following approaches:

1. If `fix-ledger.yaml` exists and lists files changed in the most recent fix round, use that list.
2. Otherwise, compare against the git state at the time of the previous review round using `git diff`.

### Step 4: Review for Integration Issues

For each file in the review scope, and across files that interact with each other:

1. **API contract consistency.** Where one module calls another (function calls, HTTP endpoints, message formats, database queries), verify that the caller's expectations match the callee's implementation. Check: parameter types and counts, return types, error types thrown or returned, HTTP methods and paths, request/response body shapes.
2. **Data flow integrity.** Trace data through the system from input to output. Verify that transformations are applied in the correct order, that required fields are present at each stage, and that no data is silently dropped or corrupted.
3. **Error propagation.** Verify that errors raised in one module are correctly caught, transformed, or propagated by calling modules. Check for: swallowed exceptions (bare `except:` or empty `catch`), incorrect error types, missing error handling on I/O operations, and inconsistent error response formats.
4. **Missing integration points.** Cross-reference `tasks.yaml` to verify that all tasks that should connect to each other actually do. Check for: modules that are implemented but never imported, endpoints that are defined but never registered, event handlers that are defined but never subscribed.

### Step 5: Review for Architectural Issues

1. **Pattern consistency.** Identify the patterns used across the codebase (naming conventions, module structure, error handling approach, configuration style) and flag deviations. For example, if most modules use a factory pattern but one uses direct instantiation, flag it.
2. **Duplication.** Identify code that performs the same or very similar logic in multiple places. Flag cases where shared utility functions or common abstractions should be extracted.
3. **Dependency direction.** Verify that dependency flow is consistent -- lower-level modules should not import from higher-level modules. Flag circular dependencies between modules.
4. **Configuration and secrets.** Verify that configuration values are not hardcoded. Check that secrets, API keys, and credentials are not present in source files or committed to the repository.

### Step 6: Review for Security Issues

1. **Input validation.** Verify that all external inputs (HTTP request bodies, query parameters, file uploads, environment variables) are validated before use.
2. **Injection vulnerabilities.** Check for SQL injection (string concatenation in queries), command injection (unescaped user input in shell commands), and cross-site scripting (unescaped user input in HTML output).
3. **Authentication and authorization.** If the PRD specifies auth requirements, verify that protected endpoints or resources check authentication and authorization before processing.
4. **Sensitive data exposure.** Check for logging of sensitive data (passwords, tokens, PII), inclusion of secrets in error messages, and overly verbose error responses in production configurations.

### Step 7: Review for Code Quality

1. **Dead code.** Flag functions, classes, variables, and imports that are defined but never used.
2. **Unused imports.** Flag imports that are not referenced in the file.
3. **Missing error handling.** Flag I/O operations (file access, network calls, database queries) that do not have error handling.
4. **Naming clarity.** Flag variables, functions, or classes with names that are ambiguous, misleading, or do not convey their purpose. Only flag names that would cause confusion -- do not enforce a specific naming convention unless the project has one.

### Step 8: Classify and Write Issues

For each issue found, assign a severity and category:

**Severity levels:**

- **`CRITICAL`** -- Broken functionality, missing integration between components, security vulnerabilities, data loss risk. These must be fixed before the project can be considered functional.
- **`MAJOR`** -- Architectural inconsistency, significant code duplication, missing error handling on important paths, broken API contracts between producers and consumers. These should be fixed but do not prevent basic functionality.
- **`MINOR`** -- Naming issues, style inconsistencies, missing documentation comments, minor code smells. These are recorded but do not trigger fix rounds.

**Categories:**

- `integration` -- Issues with how components connect and communicate.
- `architecture` -- Issues with code structure, patterns, and dependency flow.
- `security` -- Vulnerabilities, sensitive data exposure, missing input validation.
- `duplication` -- Duplicated logic that should be consolidated.
- `error-handling` -- Missing, incorrect, or inconsistent error handling.
- `style` -- Naming, formatting, documentation, and code clarity.

For each issue, identify which tasks from `tasks.yaml` are related by matching the affected file against the `files_affected` fields.

Write the issues to `<phase-path>/reviews/round-N.yaml` with the following structure:

```yaml
# Code review round N
# Phase: <phase-path>
# Scope: <full | changed>
# Generated by: /jms-plan-code-review

issues:
  - id: "ISSUE-001"
    severity: "CRITICAL"
    category: "integration"
    file: "src/auth/handler.py"
    line: 42
    description: "The login endpoint calls UserService.authenticate() with two arguments (email, password) but the method signature expects three arguments (email, password, remember_me). This will raise a TypeError at runtime."
    suggestion: "Add the missing remember_me parameter to the login endpoint call, defaulting to False if the request body does not include it."
    related_tasks:
      - "TASK-003"
      - "TASK-005"

  - id: "ISSUE-002"
    severity: "MAJOR"
    category: "error-handling"
    file: "src/api/routes.py"
    line: 87
    description: "Database query in get_user_profile() has no exception handling. If the database is unreachable, the unhandled exception will return a 500 error with a stack trace."
    suggestion: "Wrap the database call in a try/except block that catches the database connection error and returns a structured error response."
    related_tasks:
      - "TASK-007"

summary:
  total_issues: 2
  critical: 1
  major: 1
  minor: 0
  by_category:
    integration: 1
    architecture: 0
    security: 0
    duplication: 0
    error-handling: 1
    style: 0
```

**Field requirements:**

- `id` -- Format: `ISSUE-NNN` with zero-padded three-digit number, starting at `ISSUE-001` and incrementing sequentially within this round.
- `severity` -- One of: `CRITICAL`, `MAJOR`, `MINOR`.
- `category` -- One of: `integration`, `architecture`, `security`, `duplication`, `error-handling`, `style`.
- `file` -- The file path relative to the project root.
- `line` -- The line number where the issue occurs. If the issue spans multiple lines, use the first line. If the exact line cannot be determined, use `0`.
- `description` -- A clear, specific description of the problem. Must include enough context that a developer can understand the issue without reading the file.
- `suggestion` -- A concrete suggestion for how to resolve the issue.
- `related_tasks` -- A list of task IDs from `tasks.yaml` that are related to this issue (based on `files_affected` overlap). Empty list if no task directly relates.
- `summary` -- A counts section at the end of the file summarising the findings.

### Step 9: Report

Present a summary to the caller:

- The review round number (N).
- The scope used (`full` or `changed`).
- Total issues found, broken down by severity: N CRITICAL, N MAJOR, N MINOR.
- Breakdown by category.
- The output file path (`<phase-path>/reviews/round-N.yaml`).
- If CRITICAL or MAJOR issues were found, note that `/jms-plan-fix` should be run to address them.
- If only MINOR issues were found (or no issues at all), note that the codebase is ready for summary generation.

## Guidelines

- **Focus on correctness, not style.** This is a code review for functional correctness, integration integrity, and security. Do not raise issues about code formatting, bracket placement, or whitespace unless the project has an enforced style guide and the code violates it.
- **Every issue must be specific.** Vague findings like "error handling could be improved" are not acceptable. Specify the file, line, what is wrong, and what should change.
- **Do not duplicate prior findings.** Read previous review rounds and the fix ledger before writing issues. If an issue was raised in a prior round and has been fixed, do not re-raise it. If it was raised and not fixed, do not duplicate it -- it will still appear in the prior round file.
- **Severity must be honest.** Do not inflate severity to force fixes. A naming issue is MINOR, not MAJOR. A missing null check on an optional field that causes a crash is CRITICAL, not MINOR.
- **Related tasks help trace accountability.** Always link issues back to the tasks that produced the affected code. This helps the fix agent understand the original intent.
- **The review must stand alone.** A reader should be able to understand every issue in the review file without needing to read the source code, the PRD, or the task list. Include enough context in each issue description.

# jms-plan-prd-review

Critically evaluate a Product Requirements Document (PRD) against the original prompt that produced it. This skill sits between `jms-plan-prd` (which generates the PRD) and `jms-plan-task-breakdown` (which converts the approved PRD into tasks). Its purpose is to catch gaps, contradictions, ambiguities, and feasibility issues before any implementation work begins.

## Usage

```text
/jms-plan-prd-review [phase-path]
```

**Argument:**

- `[phase-path]` -- (Optional) Path to the phase directory (e.g. `.plans/2026/03/01/01-auth-implementation`). This directory must contain both `prompt.md` and `prd.md`.

If omitted, the skill lists the 5 most recent phase directories and prompts the user to select one (see Step 1).

**Directory convention:**

```text
<phase-path>/
├── prompt.md       ← input (original requirements)
├── prd.md          ← input (PRD generated by jms-plan-prd)
├── prd-review.md   ← output (generated by this skill)
├── logs/           ← created by jms-plan-phase-new
├── reviews/        ← created by jms-plan-phase-new
└── state.yaml      ← created by jms-plan-phase-new
```

## Instructions

### Step 0: Determine Repository Root

Run `git rev-parse --show-toplevel` to determine the repository root. All `.plans/` references in this skill resolve relative to this root (i.e. `<repo-root>/.plans/`). If the command fails (not a git repository), stop and tell the user this skill requires a git repository.

### Step 1: Resolve Phase Path

If a phase path argument was provided, use it directly. Verify that the directory exists and contains both `prompt.md` and `prd.md`. If the directory does not exist or either file is missing, stop and tell the user. If `prd.md` is missing, remind them to run `/jms-plan-prd` first.

If no argument was provided:

1. Scan `<repo-root>/.plans/` recursively for directories matching the pattern `NN-*` (where NN is a two-digit number) that contain both `prompt.md` and `prd.md`.
2. Sort by path in reverse lexicographic order (most recent date and highest increment first).
3. Display the 5 most recent phase directories and ask the user to select one.
4. If no qualifying phase directories are found, stop and tell the user to run `/jms-plan-prd` first to generate a PRD.

### Step 2: Read Source Documents

Read both files in full:

1. `<phase-path>/prompt.md` -- the original requirements prompt.
2. `<phase-path>/prd.md` -- the PRD generated by `/jms-plan-prd`.

### Step 3: Coverage Assessment

Compare the PRD against the original prompt line by line. For every point, requirement, goal, constraint, and scope boundary stated in the prompt, determine whether the PRD addresses it. Build a coverage matrix:

| Prompt Item | PRD Section | Covered? | Notes |
|---|---|---|---|
| *Each distinct requirement or point from the prompt* | *Where in the PRD it appears* | Yes / Partial / No | *What is missing or incomplete* |

A prompt item is **covered** if the PRD contains a requirement (with ID) and at least one acceptance criterion that directly addresses it. A prompt item is **partially covered** if it is mentioned but lacks a concrete requirement ID or testable acceptance criterion. A prompt item is **not covered** if the PRD does not address it at all.

### Step 4: Requirement-Level Evaluation

For each requirement in the PRD (`REQ-NNN`), evaluate the following:

1. **Implementability** -- "Could a developer implement this requirement without further questions?" If no, flag the specific ambiguity.
2. **Testability** -- Are the acceptance criteria concrete and objectively verifiable? Flag vague criteria (e.g. "the system should be fast" without a measurable threshold).
3. **Completeness** -- Does the requirement cover normal operation, edge cases, and error scenarios? Flag missing edge cases.
4. **Consistency** -- Does this requirement contradict any other requirement in the PRD? Flag contradictions with the specific conflicting requirement IDs.

### Step 5: Structural Analysis

Evaluate the PRD as a whole for:

1. **Contradiction detection** -- Identify any requirements that conflict with each other. For each contradiction, cite both requirement IDs and explain the conflict.
2. **Ambiguity flags** -- Identify terms, phrases, or requirements that are open to multiple interpretations. For each ambiguity, explain why it is ambiguous and what clarification is needed.
3. **Missing edge cases or error scenarios** -- Identify requirements that describe the happy path but do not account for failure modes, invalid input, boundary conditions, or concurrency issues.
4. **Feasibility concerns** -- Identify requirements that may be technically infeasible, unreasonably expensive, or contradicted by stated constraints (e.g. a performance requirement that conflicts with a technology choice).
5. **Scope consistency** -- Verify that the in-scope and out-of-scope sections in the PRD align with what the prompt stated. Flag any prompt items that appear in the PRD's out-of-scope section but were requested in the prompt, or PRD requirements that go beyond what the prompt asked for.

### Step 6: Determine Verdict

Based on the analysis from Steps 3-5, assign one of three verdicts:

**`PASS`** -- The PRD is ready to proceed to Task Breakdown. All of the following must be true:

- Every prompt item is fully covered by at least one PRD requirement.
- No contradictions exist between requirements.
- No requirement has ambiguities that would block implementation.
- All acceptance criteria are testable and specific.
- No critical feasibility concerns.

**`REVISE`** -- The PRD needs changes before proceeding. One or more of the following are true:

- Prompt items are partially covered or not covered.
- Contradictions exist between requirements.
- Requirements contain ambiguities that would block implementation.
- Acceptance criteria are vague or untestable.
- Edge cases or error scenarios are missing for critical requirements.
- Scope boundaries do not align with the prompt.

When issuing a `REVISE` verdict, provide specific, actionable feedback for each issue. Each piece of feedback must state: what is wrong, which requirement or prompt item is affected, and exactly what needs to change. Do not give vague suggestions like "add more detail" -- instead say "REQ-003 acceptance criteria do not specify the expected HTTP status code for invalid input; add a criterion such as 'Returns 400 Bad Request with a JSON error body when the email field is missing'".

**`ESCALATE`** -- The review cannot be completed or the issues are severe enough that human judgment is required. Use this verdict when:

- The prompt itself is fundamentally unclear or contradictory, making it impossible to assess PRD coverage.
- The PRD and prompt describe substantially different projects.
- There are feasibility concerns that require domain expertise or business decisions to resolve.
- Multiple interrelated contradictions make targeted revision impractical.

When issuing an `ESCALATE` verdict, explain clearly why automated resolution is not possible and what specific decisions or clarifications are needed from the user.

### Step 8: Human-in-the-Loop Gate (Optional)

After generating the review document, check whether the user wants to approve the PRD regardless of the automated verdict. Present the following prompt:

```text
PRD review complete. Verdict: <VERDICT>

<If PASS>  The PRD passed automated review and is ready for task breakdown.
<If REVISE>  The PRD has issues that should be addressed before proceeding.
<If ESCALATE>  The review flagged issues requiring your input.

Would you like to:
  1. Accept the verdict and proceed accordingly
  2. Override the verdict and approve the PRD for task breakdown
  3. Review the full report before deciding
```

If the user selects option 2, note the override in the review document by appending:

```markdown
## Human Override

Verdict overridden to **PASS** by user. Original verdict: <ORIGINAL_VERDICT>.
```

If the user selects option 3, display the contents of `prd-review.md` and then re-present options 1 and 2.

### Step 9: Report

Report a brief summary to the user:

- The verdict (`PASS`, `REVISE`, or `ESCALATE`)
- Coverage statistics (X of Y prompt items fully covered)
- Number of structural issues found (contradictions, ambiguities, missing edge cases, feasibility concerns, scope inconsistencies)
- Number of action items (if any)
- Next step guidance:
  - If `PASS`: "Run `/jms-plan-task-breakdown <phase-path>` to generate the task list."
  - If `REVISE`: "Update `prd.md` to address the action items, then re-run `/jms-plan-prd-review <phase-path>`."
  - If `ESCALATE`: "Review the flagged issues and update `prompt.md` or `prd.md` as needed, then re-run `/jms-plan-prd-review <phase-path>`."

## Guidelines

- **Be critical, not adversarial.** The goal is to catch real problems, not to generate busywork. Do not flag issues that are purely stylistic or that would not block a competent developer.
- **Specificity over volume.** Five specific, actionable findings are more valuable than twenty vague observations. Every issue flagged must include a concrete suggestion for resolution.
- **Respect settled decisions.** If the prompt specifies a technology, approach, or constraint, do not flag it as a feasibility concern unless it genuinely conflicts with another requirement.
- **REVISE feedback must be actionable.** Each action item must describe the exact change needed. "Add more acceptance criteria" is not actionable. "REQ-005 needs an acceptance criterion covering the case where the API returns a 429 rate-limit response" is actionable.
- **ESCALATE is a last resort.** Use it only when the issues cannot be resolved by revising the PRD alone and require human judgment or clarification of the original prompt.
- **The review must stand alone.** A reader should be able to understand every finding in the review document without needing to cross-reference the PRD or prompt.

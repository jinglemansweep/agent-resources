# jms-plan-summary

Generate a comprehensive summary report for a completed phase. This is a standalone, user-invocable skill primarily called by the `jms-plan-execute` orchestrator at the end of the pipeline. It reads all logs, review reports, fix records, task definitions, and project context to produce a single `summary.md` that captures everything that happened during the phase.

## Usage

```text
/jms-plan-summary [phase-path]
```

**Argument:**

- `[phase-path]` -- (Optional) Path to the phase directory (e.g. `.plans/2026/03/01/01-auth-implementation`).

If omitted, the skill lists the 5 most recent phase directories and prompts the user to select one (see Step 1).

**Directory convention:**

```text
<phase-path>/
├── prompt.md           <- input (original requirements)
├── prd.md              <- input (approved PRD)
├── prd-review.md       <- input (PRD review, optional)
├── tasks.yaml          <- input (task list)
├── task-review.md      <- input (task review, optional)
├── state.yaml          <- input (execution state)
├── fix-ledger.yaml     <- input (fix history, optional)
├── logs/
│   ├── task-001.md     <- input (per-task execution logs)
│   ├── task-002.md
│   └── ...
├── reviews/
│   ├── round-1.yaml    <- input (code review rounds)
│   ├── round-2.yaml
│   └── ...
└── summary.md          <- output (generated by this skill)
```

## Instructions

### Step 0: Determine Repository Root

Run `git rev-parse --show-toplevel` to determine the repository root. All `.plans/` references in this skill resolve relative to this root (i.e. `<repo-root>/.plans/`). If the command fails (not a git repository), stop and tell the user this skill requires a git repository.

### Step 1: Resolve Phase Path

If a phase path argument was provided, use it directly. Verify that the directory exists. If it does not exist, stop and tell the user.

If no argument was provided:

1. Scan `<repo-root>/.plans/` recursively for directories matching the pattern `NN-*` (where NN is a two-digit number).
2. Sort by path in reverse lexicographic order (most recent date and highest increment first).
3. Display the 5 most recent phase directories and ask the user to select one.
4. If no qualifying phase directories are found, stop and tell the user.

### Step 2: Read All Phase Artifacts

Read every available artifact from the phase directory. Not all files may exist -- handle missing files gracefully and note their absence in the summary.

**Required files (stop if missing):**

- `<phase-path>/prd.md` -- The Product Requirements Document.
- `<phase-path>/tasks.yaml` -- The task list.

**Optional files (note absence but continue):**

- `<phase-path>/prompt.md` -- The original requirements prompt.
- `<phase-path>/prd-review.md` -- The PRD review.
- `<phase-path>/task-review.md` -- The task list review.
- `<phase-path>/state.yaml` -- The execution state tracker.
- `<phase-path>/fix-ledger.yaml` -- The fix round history.
- All files matching `<phase-path>/logs/task-*.md` -- Per-task execution logs.
- All files matching `<phase-path>/reviews/round-*.yaml` -- Code review round files.

### Step 3: Extract Project Overview

From `prd.md`, extract:

1. The project overview section (what the project is, why it exists, who it is for).
2. The goals section.
3. A count of total requirements (functional + non-functional).

### Step 4: Extract Task Status

From `tasks.yaml`, build a complete task inventory. For each task, record:

- Task ID, title, and suggested role.
- Estimated complexity.
- Requirements addressed.

If `state.yaml` exists, cross-reference it to determine the actual execution status of each task (completed, failed, blocked, skipped). If `state.yaml` does not exist, note that execution state is unavailable.

If per-task logs exist (`logs/task-NNN.md`), read each log to extract:

- Whether the task succeeded or failed.
- Files created or modified.
- Any issues encountered during execution.
- The agent role that executed the task.
- Retry count (if the task required retries).

### Step 5: Extract File Manifest

Build a complete list of all files created or modified during the phase. Sources for this list:

1. The `files_affected` field from each task in `tasks.yaml`.
2. Files mentioned in per-task logs.
3. Files mentioned in the fix ledger.

Deduplicate the list and categorize each file as: created (new file), modified (existing file changed), or unknown (if the creation/modification status cannot be determined).

### Step 6: Extract Review History

If review round files exist (`reviews/round-*.yaml`), read each one and extract:

- The round number and scope (full or changed).
- Total issues found, broken down by severity (CRITICAL, MAJOR, MINOR).
- Breakdown by category.
- A list of the most significant issues (all CRITICAL, and MAJOR issues up to a reasonable limit).

If the fix ledger exists (`fix-ledger.yaml`), cross-reference each review round with the corresponding fix round to determine:

- How many issues were fixed.
- How many issues were deferred.
- Whether any regressions were introduced.
- The exit reason for each fix round.

Calculate totals across all rounds: total issues found, total fixed, total deferred, total remaining.

### Step 7: Extract Known Issues and Deferred Items

Compile a list of all known issues and deferred items from:

1. **Deferred issues** from `fix-ledger.yaml` -- issues flagged as `DEFERRED` with their explanations.
2. **Remaining issues** from the final review round -- any CRITICAL or MAJOR issues that were not fixed (because the fix loop reached its maximum rounds or because regression detection stopped the loop).
3. **MINOR issues** from all review rounds -- these are never fixed but should be documented for future reference.
4. **Blocked tasks** from `state.yaml` or task logs -- tasks that could not be completed.

### Step 8: Extract Decisions

Scan per-task logs and review files for decisions made during execution:

1. Technology choices (libraries selected, frameworks used, tools configured).
2. Trade-offs (performance vs. readability, completeness vs. time, strict vs. permissive validation).
3. Deviations from the PRD (any requirements that were implemented differently than specified, with rationale).
4. Assumptions made by implementing agents.

If no explicit decisions can be identified from the logs, note "No explicit decisions recorded in execution logs."

### Step 9: Determine Next Steps

Based on the current state of the phase, suggest concrete next steps:

1. **If deferred issues exist**, list them with a recommendation to address them in a follow-up phase.
2. **If MINOR issues remain**, suggest a cleanup pass.
3. **If blocked tasks exist**, suggest what is needed to unblock them.
4. **If all tasks completed and no issues remain**, suggest next actions such as: manual testing, deployment, documentation review, or user acceptance testing.
5. **If the PRD has requirements not covered by completed tasks**, flag the coverage gap.

Write `<phase-path>/summary.md` with the following structure:


### Step 11: Verify

Read the written `summary.md` back to confirm it was written correctly and contains all expected sections.

### Step 12: Report

Present a brief summary to the user:

- The output file path (`<phase-path>/summary.md`).
- Task execution overview: N completed, N failed, N blocked out of N total.
- Review overview: N review rounds, N total issues found, N fixed, N deferred, N remaining.
- Number of known issues and deferred items.
- Whether any suggested next steps were generated.

## Guidelines

- **Accuracy over completeness.** Only include information that can be verified from the phase artifacts. If a log file is missing, note the gap rather than guessing at the task outcome.
- **The summary must stand alone.** A reader should be able to understand the full state of the phase from `summary.md` without reading any other file in the phase directory.
- **Do not editorialize.** Report what happened factually. Do not add opinions about code quality, developer performance, or process effectiveness. Stick to the data from the artifacts.
- **Handle missing files gracefully.** Not every phase will have all artifacts. If `fix-ledger.yaml` does not exist, the review rounds section should note that no fix rounds were conducted. If no task logs exist, note that per-task execution details are unavailable.
- **Deferred items are important.** Every deferred issue should be prominently documented so it is not forgotten. Include the full deferral reason so a future developer understands why it was not addressed.
- **Next steps should be actionable.** "Improve the code" is not a next step. "Address ISSUE-012 (MAJOR) by redesigning the session management interface to support both cookie-based and token-based authentication" is a next step.
- **Keep it concise for large phases.** If a phase has dozens of tasks and hundreds of issues, use tables and summaries rather than listing every detail inline. The goal is a readable overview, not a data dump.

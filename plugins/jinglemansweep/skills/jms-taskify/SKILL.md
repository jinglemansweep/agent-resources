# jms-taskify

Generate a detailed, actionable task list from an existing plan. This is the second stage of a two-step workflow — it consumes the outputs of `jms-plan` and produces a comprehensive task list that can drive implementation directly.

## Usage

```text
/jms-taskify <plan-dir>
```

**Argument:**

- `<plan-dir>` — Path to the plan directory (e.g. `.plans/20260115/01-initial-implementation`). This directory must already contain `plan.md` and should contain `research.md` and `issues.md`.

**Directory convention:**

```text
<plan-dir>/
├── prompt.md       ← original requirements
├── research.md     ← research findings (input)
├── plan.md         ← implementation plan (input)
├── issues.md       ← issues and risks (input)
└── tasks.md        ← output (generated by this skill)
```

## Instructions

### Step 1: Read Inputs

Read the following files from `<plan-dir>/`:

1. `plan.md` — **required**. If missing, stop and tell the user to run `jms-plan` first.
2. `research.md` — read if present, skip if not.
3. `issues.md` — read if present, skip if not.
4. `prompt.md` — read if present, for additional context.

### Step 2: Analyze

Study all inputs and identify:

- Every component, file, and system described in the plan
- The dependency order between components (what must be built first)
- Research findings that affect implementation (version constraints, API details, compatibility notes)
- Issues that require mitigation steps or validation tasks
- The logical grouping of work (by component, by layer, by feature — whichever best fits the project)

### Step 3: Generate the Task List

Generate `<plan-dir>/tasks.md` structured as follows:

```markdown
# Task List

> Source: `<plan-dir>/plan.md`

## <Group Name>
<!-- A logical grouping of related work -->

### <Subgroup Name>
<!-- A narrower grouping within the parent. Omit if the group
     doesn't need further subdivision. -->

- [ ] **Task title** — Detailed description of exactly what needs to be done. Include: the file(s) to create or modify, the specific content or logic required, any values/constants/paths to use, and how to verify the task is complete. Reference research findings or issues where relevant.
  - [ ] Subtask if the parent task has distinct sub-steps that should be tracked independently
  - [ ] ...
- [ ] **Next task** — ...

### <Next Subgroup>
- [ ] ...

## <Next Group>
...
```

Write the task list to `<plan-dir>/tasks.md`.

### Step 4: Report

Report a brief summary to the user confirming what was written:

- Total number of groups
- Total number of tasks (including subtasks)
- Any issues from `issues.md` that were incorporated as tasks

## Guidelines

### Task Detail

Each task must contain enough information for an implementing agent to execute it **without reading any other file**. This means:

- **Specify file paths** — not "create the config file" but "create `src/config/schema.py`"
- **Specify content** — not "add the mapping" but "add a field `name` of type `string` with validation for non-empty values"
- **Specify behaviour** — not "handle errors" but "if the input is invalid, raise a `ValueError` with a message describing which field failed validation"
- **Specify verification** — not "test it works" but "run the test suite with `pytest tests/` and verify all tests pass"

### Task Granularity

- Each task should represent a single, focused unit of work — something that results in one logical change (a file created, a function written, a config entry added).
- If a task description exceeds ~3 sentences, consider splitting it into subtasks.
- Subtasks should be used when a parent task has distinct steps that benefit from independent tracking.

### Ordering & Dependencies

- Order groups so that foundational/infrastructure work comes before work that depends on it.
- Within each group, order tasks in the sequence they should be implemented.
- If a task has a hard dependency on another task in a different group, note it explicitly in the task description (e.g. "Requires: Config Schema group to be complete").

### Incorporating Issues

- For each open question in `issues.md`, either:
  - Resolve it by making a reasonable default choice and noting it in the task description, or
  - Add a task to investigate/decide before proceeding
- For each risk, add a validation or mitigation task if appropriate.
- For potential blockers, add investigation tasks early in the ordering.

### What NOT to Include

- Do not include architecture explanations, rationale, or design discussion — that belongs in `plan.md`.
- Do not include research findings — that belongs in `research.md`.
- Do not repeat the file manifest from the plan — instead, the tasks themselves should reference the files.
- Do not add project management overhead tasks (e.g. "review plan", "hold kickoff meeting").

# jp-plan-prd

Generate a Product Requirements Document (PRD) from a prompt file. The PRD produced here feeds into `jp-plan-prd-review` for validation, then `jp-plan-task-breakdown` for conversion into an actionable task list.

## Usage

```text
/jp-plan-prd [phase-path]
```

**Argument:**

- `[phase-path]` -- (Optional) Path to the phase directory (e.g. `.plans/2026/03/01/01-auth-implementation`). This directory must already contain a `prompt.md` file.

If omitted, the skill lists the 5 most recent phase directories and prompts the user to select one (see Step 1).

**Directory convention:**

```text
<phase-path>/
├── prompt.md       ← input (must exist)
├── prd.md          ← output (generated by this skill)
├── logs/           ← created by jp-plan-new
├── reviews/        ← created by jp-plan-new
└── state.yaml      ← created by jp-plan-new
```

## Instructions

### Step 0: Determine Repository Root

Run `git rev-parse --show-toplevel` to determine the repository root. All `.plans/` references in this skill resolve relative to this root (i.e. `<repo-root>/.plans/`). If the command fails (not a git repository), stop and tell the user this skill requires a git repository.

### Step 1: Resolve Phase Path

If a phase path argument was provided, use it directly. Verify that the directory exists and contains `prompt.md`. If the directory or file does not exist, stop and tell the user.

If no argument was provided:

1. Scan `<repo-root>/.plans/` recursively for directories matching the pattern `NN-*` (where NN is a two-digit number) that contain a `prompt.md` file.
2. Sort by path in reverse lexicographic order (most recent date and highest increment first).
3. Display the 5 most recent phase directories and ask the user to select one.
4. If no phase directories are found, stop and tell the user to run `/jp-plan-new` first.

### Step 2: Read the Prompt

Read `<phase-path>/prompt.md` in full.

### Step 3: Analyze

Analyze the prompt thoroughly. Identify:

- The project's goals and desired outcomes
- Functional capabilities the system must provide
- Non-functional qualities the system must exhibit (performance, security, accessibility, reliability)
- Technical constraints and assumptions (stack, infrastructure, compatibility)
- What is explicitly in scope and out of scope
- External references, dependencies, or integrations mentioned

If the prompt contains explicit decisions (e.g. a decisions table, a specified tech stack), respect them -- do not second-guess or re-open settled choices.

### Step 4: Clarify (Only If Necessary)

Review the analysis for critical gaps -- information without which a meaningful PRD cannot be produced and no reasonable default exists. Examples of critical gaps:

- The prompt describes a system but gives no indication of what it should do (no goals, no requirements)
- A core technology choice is referenced but ambiguous (e.g. "use a database" with no hint of relational vs. document vs. key-value)

If critical gaps exist, ask the user targeted clarifying questions. Keep the number of questions to an absolute minimum. Do not ask about:

- Details that can be reasonably inferred or defaulted
- Non-functional requirements that have standard defaults (e.g. "should we have error handling?" -- yes, always)
- Implementation specifics that belong in the task breakdown stage

If no critical gaps exist, proceed without asking questions.

### Step 5: Generate the PRD

Generate `<phase-path>/prd.md` structured as follows:

```markdown
# Product Requirements Document

> Source: `<phase-path>/prompt.md`

## Project Overview

<!-- 2-4 sentences describing what this project delivers, why it exists, and
     who it is for. -->

## Goals

<!-- Bulleted list of the specific, measurable goals this project aims to
     achieve. Each goal should describe an outcome, not a task. -->

- ...

## Functional Requirements

<!-- Each requirement has a unique ID, a title, a description, and at least
     one acceptance criterion. Requirements should be specific enough that a
     developer can implement them without further questions. -->

### REQ-001: <Requirement Title>

**Description:** <What the system must do>

**Acceptance Criteria:**

- [ ] <Criterion 1 -- a testable condition that proves the requirement is met>
- [ ] <Criterion 2>

### REQ-002: <Requirement Title>

...

## Non-Functional Requirements

<!-- Requirements covering performance, security, accessibility, reliability,
     scalability, and other quality attributes. Same ID and acceptance
     criteria format as functional requirements. -->

### REQ-NNN: <Requirement Title>

**Category:** <Performance | Security | Accessibility | Reliability | Scalability | Maintainability>

**Description:** <What quality attribute the system must exhibit>

**Acceptance Criteria:**

- [ ] <Criterion>

## Technical Constraints and Assumptions

<!-- Constraints are hard limits (e.g. "must run on Python 3.11+").
     Assumptions are beliefs taken as true for planning purposes
     (e.g. "users have modern browsers"). -->

### Constraints

- ...

### Assumptions

- ...

## Scope

### In Scope

- ...

### Out of Scope

- ...

## Suggested Tech Stack

<!-- Based on the prompt's explicit choices and the requirements above,
     list the recommended technologies, frameworks, and tools. If the
     prompt already specifies a stack, list it here and note it as
     "specified by prompt". Only suggest alternatives if the prompt
     leaves the choice open. -->

| Layer | Technology | Rationale |
|-------|-----------|-----------|
| ... | ... | ... |
```

**Requirement ID numbering:**

- Functional requirements start at `REQ-001` and increment sequentially.
- Non-functional requirements continue the same sequence (e.g. if the last functional requirement is `REQ-012`, the first non-functional requirement is `REQ-013`).
- Every requirement -- functional and non-functional -- must have at least one acceptance criterion.

### Step 6: Report

Report a brief summary to the user confirming what was written to `<phase-path>/prd.md`:

- Total number of functional requirements
- Total number of non-functional requirements
- Total number of acceptance criteria across all requirements
- Any assumptions that were made where the prompt was ambiguous
- A reminder to run `/jp-plan-prd-review <phase-path>` to validate the PRD before proceeding

## Guidelines

- **This is a PRD, not an implementation plan or task list.** Focus on what the system must do and what qualities it must have. Do not describe how to build it, do not produce task lists, and do not include code examples unless essential to define a requirement.
- **Requirements must be testable.** Each acceptance criterion should describe a condition that can be objectively verified -- not vague statements like "the system should be fast" but specific thresholds like "API responses must complete within 200ms under normal load".
- **Be comprehensive but not padded.** Cover all requirements implied by the prompt. Do not invent requirements that are not supported by the prompt. If the prompt is narrow, the PRD should be narrow.
- **Respect the prompt's decisions.** If the prompt specifies a technology, framework, or approach, include it as a constraint -- do not propose alternatives unless asked.
- **Scope boundaries matter.** Explicitly listing what is out of scope prevents scope creep during implementation. Derive out-of-scope items from the prompt's context and constraints.
- **The PRD should stand alone.** A reader should be able to understand the full scope of the project from the PRD without needing to read the original prompt.

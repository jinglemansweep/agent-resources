# jp-task-list

Convert an approved Product Requirements Document (PRD) into a structured YAML task list. This skill replaces `jms-taskify` and produces `tasks.yaml` instead of `tasks.md`. It sits between `jp-prd-review` (which validates the PRD) and `jp-task-review` (which validates the task list). The output drives `jp-execute` for implementation.

## Usage

```text
/jp-task-list [phase-path]
```

**Argument:**

- `[phase-path]` -- (Optional) Path to the phase directory (e.g. `.plans/2026/03/01/01-auth-implementation`). This directory must contain an approved `prd.md`.

If omitted, the skill lists the 5 most recent phase directories and prompts the user to select one (see Step 1).

**Directory convention:**

```text
<phase-path>/
├── prompt.md       <- input (original requirements)
├── prd.md          <- input (approved PRD from jp-prd)
├── prd-review.md   <- input (review from jp-prd-review, optional)
├── tasks.yaml      <- output (generated by this skill)
├── logs/           <- created by jp-plan
├── reviews/        <- created by jp-plan
└── state.yaml      <- created by jp-plan
```

## Instructions

### Step 0: Determine Repository Root

Run `git rev-parse --show-toplevel` to determine the repository root. All `.plans/` references in this skill resolve relative to this root (i.e. `<repo-root>/.plans/`). If the command fails (not a git repository), stop and tell the user this skill requires a git repository.

### Step 1: Resolve Phase Path

If a phase path argument was provided, use it directly. Verify that the directory exists and contains `prd.md`. If the directory does not exist or `prd.md` is missing, stop and tell the user. If `prd.md` is missing, remind them to run `/jp-prd` first.

If no argument was provided:

1. Scan `<repo-root>/.plans/` recursively for directories matching the pattern `NN-*` (where NN is a two-digit number) that contain a `prd.md` file.
2. Sort by path in reverse lexicographic order (most recent date and highest increment first).
3. Display the 5 most recent phase directories and ask the user to select one.
4. If no qualifying phase directories are found, stop and tell the user to run `/jp-prd` first to generate a PRD.

### Step 2: Read the PRD

Read `<phase-path>/prd.md` in full.

Also read `<phase-path>/prd-review.md` if it exists, to understand any review feedback or constraints that were raised. Also read `<phase-path>/prompt.md` if it exists, for additional context.

### Step 3: Extract Requirements

Parse the PRD and build a complete list of all requirements:

1. Collect every requirement ID (`REQ-NNN`) from both the Functional Requirements and Non-Functional Requirements sections.
2. For each requirement, record: the ID, the title, the description, and all acceptance criteria.
3. Note any technical constraints, assumptions, and scope boundaries from the PRD.
4. Note the suggested tech stack if present.

### Step 4: Design the Task Graph

Plan the task breakdown before generating output. For each requirement:

1. Identify the concrete units of work needed to satisfy all acceptance criteria.
2. Determine dependencies between tasks -- what must be built before what.
3. Identify shared infrastructure work (project setup, configuration, CI, Docker) that multiple requirements depend on.
4. Identify test-writing tasks as separate, explicit tasks (not folded into implementation tasks).
5. Identify documentation tasks where the PRD or requirements imply user-facing docs, API docs, or configuration guides.
6. Identify infrastructure tasks (Dockerfile, CI config, deployment scripts) where the PRD implies containerization, CI/CD, or deployment.

**Task design principles:**

- **Atomic tasks.** Each task represents one clear unit of work. A task should produce one logical change -- a module created, a function implemented, a test suite written, a config file added.
- **Dependency-driven ordering.** If task B cannot start until task A is complete, A must appear in B's `dependencies` list.
- **Full requirement coverage.** Every `REQ-NNN` from the PRD must appear in at least one task's `requirements` list. After designing the task graph, verify this by checking every requirement ID against the task list. If any requirement is not covered, add tasks until it is.
- **Test tasks are explicit.** Do not include "and write tests" as part of an implementation task. Create separate tasks for writing unit tests, integration tests, or end-to-end tests.
- **Infrastructure tasks come first.** Project scaffolding, configuration, and CI setup tasks should have no dependencies and appear early in the ordering.

### Step 5: Assign Roles

For each task, determine the most appropriate `suggested_role` based on the task content. The role must map to one of the existing role agents:

| Role | Domain Skill | Use When |
|------|-------------|----------|
| `python` | `role-python` | Python implementation, Python tests, Python tooling |
| `nodejs` | `role-nodejs` | Node.js/TypeScript implementation, frontend build tooling |
| `devops` | `role-devops` | Docker, CI/CD, infrastructure, deployment, shell scripts |
| `frontend` | `role-frontend` | HTML, CSS, React/Vue/Svelte, browser-side JavaScript |
| `general` | (none -- base behaviour) | Tasks that do not fit a specific domain, or span multiple domains |
| `docs` | `role-docs` | Documentation, README, changelog, API docs |
| `skills` | `role-agent-skills` | Skill authoring, SKILL.md files, plugin definitions |

If a task spans multiple domains, choose the primary domain. If genuinely ambiguous, use `general`.

### Step 6: Estimate Complexity

For each task, assign an `estimated_complexity` value:

- **`small`** -- A single file change, a simple function, a config entry, a straightforward test. Can be completed in isolation with minimal context.
- **`medium`** -- Multiple related changes, a module with several functions, integration logic, a test suite covering multiple scenarios. Requires understanding of adjacent components.
- **`large`** -- A core system component, complex business logic, significant integration work, a comprehensive test suite. Requires deep understanding of the system architecture.

### Step 7: Generate tasks.yaml

Perform a topological sort of the task graph based on the `dependencies` field. Tasks with no dependencies come first. Among tasks at the same dependency level, order by: infrastructure before implementation, implementation before tests, tests before documentation.

Generate `<phase-path>/tasks.yaml` with the following structure. Every task must include all fields shown below:

```yaml
# Task list generated from PRD: <phase-path>/prd.md
# Generated by: /jp-task-list

tasks:
  - id: "TASK-001"
    title: "Short descriptive title"
    description: "What needs to be done and why. Include enough detail that an implementing agent can execute this task without reading any other file."
    requirements:
      - "REQ-001"
    dependencies: []
    suggested_role: "devops"
    acceptance_criteria:
      - "Criterion 1 -- a testable condition that proves the task is complete"
      - "Criterion 2"
    estimated_complexity: "small"
    files_affected:
      - "path/to/file"

  - id: "TASK-002"
    title: "Another task"
    description: "Detailed description."
    requirements:
      - "REQ-001"
      - "REQ-002"
    dependencies:
      - "TASK-001"
    suggested_role: "python"
    acceptance_criteria:
      - "Criterion"
    estimated_complexity: "medium"
    files_affected:
      - "src/module.py"
```

**Field requirements:**

- `id` -- Format: `TASK-NNN` with zero-padded three-digit number, starting at `TASK-001` and incrementing sequentially.
- `title` -- A short, descriptive title in imperative form (e.g. "Create database schema", "Write unit tests for auth module").
- `description` -- Detailed description of what needs to be done and why. Must contain enough information for an implementing agent to execute without reading other files. Include specific file paths, content expectations, and behaviour.
- `requirements` -- List of PRD requirement IDs (`REQ-NNN`) that this task addresses. Must contain at least one entry.
- `dependencies` -- List of task IDs (`TASK-NNN`) that must complete before this task can start. Empty list if no dependencies.
- `suggested_role` -- One of: `python`, `nodejs`, `devops`, `frontend`, `general`, `docs`, `skills`.
- `acceptance_criteria` -- List of testable conditions that prove the task is complete. Must contain at least one entry.
- `estimated_complexity` -- One of: `small`, `medium`, `large`.
- `files_affected` -- List of file paths that this task will create or modify. Must contain at least one entry.

### Step 8: Validate

Before writing the file, perform the following validation checks:

1. **Requirement coverage.** Build a set of all `REQ-NNN` IDs from the PRD. Build a set of all requirement IDs referenced across all tasks. Verify every PRD requirement appears in at least one task. If any requirement is missing, add tasks to cover it.
2. **Dependency validity.** For each task, verify that every ID in its `dependencies` list refers to an existing task ID. Flag and fix any references to non-existent tasks.
3. **No circular dependencies.** Walk the dependency graph and verify it is a directed acyclic graph (DAG). If cycles exist, restructure the tasks to break them.
4. **ID uniqueness.** Verify no two tasks share the same `id`.
5. **Field completeness.** Verify every task has all required fields and no field is empty where a value is required (`requirements`, `acceptance_criteria`, `files_affected` must each have at least one entry).
6. **Role validity.** Verify every `suggested_role` value is one of: `python`, `nodejs`, `devops`, `frontend`, `general`, `docs`, `skills`.
7. **Complexity validity.** Verify every `estimated_complexity` value is one of: `small`, `medium`, `large`.

### Step 9: Write and Verify

1. Write the validated YAML to `<phase-path>/tasks.yaml`.
2. Read the file back to confirm it was written correctly and contains valid YAML with all required fields.

### Step 10: Report

Report a brief summary to the user confirming what was written to `<phase-path>/tasks.yaml`:

- Total number of tasks
- Breakdown by complexity: N small, N medium, N large
- Breakdown by role: N python, N nodejs, N devops, N frontend, N general, N docs
- Number of PRD requirements covered and total PRD requirements (should be equal)
- Maximum dependency depth (longest chain in the task graph)
- Any assumptions or decisions made during task breakdown
- A reminder to run `/jp-task-review <phase-path>` to validate the task list before proceeding to execution

## Guidelines

- **This is a task list, not a design document.** Focus on what to do, not why or how the system is designed. Architecture and rationale belong in the PRD.
- **Tasks must be self-contained.** Each task description must contain enough information for an implementing agent to execute it without reading the PRD, other tasks, or any other file. Include specific file paths, expected content, and behaviour.
- **Test tasks are first-class.** Writing tests is not a subtask of implementation -- it is a separate task with its own ID, dependencies, acceptance criteria, and role assignment.
- **Infrastructure tasks are first-class.** Project scaffolding, Docker setup, CI configuration, and deployment scripts are separate tasks, not afterthoughts appended to implementation tasks.
- **Respect the PRD's scope.** Do not add tasks for features or requirements not present in the PRD. Do not omit tasks for requirements that are in the PRD. The task list is a faithful decomposition of the PRD.
- **Dependency ordering matters.** A task that writes to a file must depend on the task that creates that file. A test task must depend on the implementation task it tests. Get the ordering right.
- **Be specific about files.** Use actual file paths (e.g. `src/auth/handler.py`), not vague references (e.g. "the auth module"). If the PRD does not specify a file structure, design a reasonable one and use it consistently across all tasks.
- **Complexity estimates should be honest.** Do not default everything to `medium`. A config file creation is `small`. A core business logic module is `medium` or `large`. A comprehensive integration test suite is `large`.

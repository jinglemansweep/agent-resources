# jp-task-review

Validate a `tasks.yaml` file against its source PRD to ensure the task list is structurally sound, correctly ordered, and fully covers all requirements. This skill sits between `jp-task-list` (which generates the task list) and `jp-execute` (which implements the tasks). Its purpose is to catch dependency errors, ordering problems, coverage gaps, and scope issues before any implementation work begins.

## Usage

```text
/jp-task-review [phase-path]
```

**Argument:**

- `[phase-path]` -- (Optional) Path to the phase directory (e.g. `.plans/2026/03/01/01-auth-implementation`). This directory must contain both `tasks.yaml` and `prd.md`.

If omitted, the skill lists the 5 most recent phase directories and prompts the user to select one (see Step 1).

**Directory convention:**

```text
<phase-path>/
├── prompt.md         <- input (original requirements)
├── prd.md            <- input (approved PRD from jp-prd)
├── reviews/prd.md    <- input (review from jp-prd-review, optional)
├── tasks.yaml        <- input (task list from jp-task-list)
├── reviews/tasks.md  <- output (generated by this skill)
├── changelog.md      <- created by jp-execute
├── reviews/          <- created by jp-plan
└── state.yaml        <- created by jp-plan
```

## Instructions

### Step 0: Determine Repository Root

Run `git rev-parse --show-toplevel` to determine the repository root. All `.plans/` references in this skill resolve relative to this root (i.e. `<repo-root>/.plans/`). If the command fails (not a git repository), stop and tell the user this skill requires a git repository.

### Step 1: Resolve Phase Path

If a phase path argument was provided, use it directly. Verify that the directory exists and contains both `tasks.yaml` and `prd.md`. If the directory does not exist or either file is missing, stop and tell the user. If `tasks.yaml` is missing, remind them to run `/jp-task-list` first. If `prd.md` is missing, remind them to run `/jp-prd` first.

If no argument was provided:

1. Scan `<repo-root>/.plans/` recursively for directories matching the pattern `NN-*` (where NN is a two-digit number) that contain both `tasks.yaml` and `prd.md`.
2. Sort by path in reverse lexicographic order (most recent date and highest increment first).
3. Display the 5 most recent phase directories and ask the user to select one.
4. If no qualifying phase directories are found, stop and tell the user to run `/jp-task-list` first to generate a task list.

### Step 2: Read Source Documents

Read both files in full:

1. `<phase-path>/tasks.yaml` -- the task list generated by `/jp-task-list`.
2. `<phase-path>/prd.md` -- the approved PRD.

Also read `<phase-path>/reviews/prd.md` if it exists, for additional context on any review feedback or constraints that were raised.

### Step 3: Parse and Inventory

Parse `tasks.yaml` and build a complete inventory:

1. Collect every task entry. For each task, record: `id`, `title`, `description`, `requirements`, `dependencies`, `suggested_role`, `acceptance_criteria`, `estimated_complexity`, and `files_affected`.
2. Build a set of all task IDs.
3. Build a set of all `REQ-NNN` IDs referenced across all tasks.
4. Parse the PRD and build a complete set of all `REQ-NNN` IDs defined in the Functional Requirements and Non-Functional Requirements sections.

### Step 4: Dependency Validation

Build the dependency graph from the `dependencies` field of each task and perform the following checks:

1. **Reference validity.** For each task, verify that every ID in its `dependencies` list refers to an existing task ID in the inventory. Record any references to non-existent task IDs.
2. **DAG validation.** Walk the dependency graph and verify it is a directed acyclic graph (DAG). Use a topological sort algorithm (e.g. Kahn's algorithm or depth-first search with cycle detection). If a cycle is found, record the exact cycle path (e.g. `TASK-003 -> TASK-007 -> TASK-003`).
3. **Orphan detection.** Identify any tasks (other than root tasks with no dependencies) that are never referenced as a dependency by any other task and do not appear to be terminal tasks (e.g. documentation or summary tasks). These may indicate missing integration points. Record them as warnings, not errors.

### Step 5: Ordering Check

Simulate execution order by performing a topological traversal of the dependency graph. At each step, track the set of files that have been "created" by completed tasks (using the `files_affected` field):

1. Process tasks in topological order. For each task, before marking it as "executed":
   - Check the task's `description` and `files_affected` for references to files that should already exist.
   - Cross-reference against the `files_affected` of all tasks in the dependency chain (direct and transitive dependencies).
   - If the task references a file that is listed in `files_affected` of another task that is NOT in its dependency chain, flag it as an ordering issue. Record the task ID, the file referenced, and which task produces that file.
2. Check that every task's `dependencies` list is sufficient -- if task B modifies a file created by task A, then A should be in B's dependency chain (directly or transitively).

### Step 6: Coverage Check

Verify that every PRD requirement is addressed by at least one task:

1. Compare the set of all `REQ-NNN` IDs from the PRD (Step 3) against the set of all requirement IDs referenced across all tasks (Step 3).
2. For each PRD requirement not covered by any task, record it as a coverage gap.
3. For each requirement ID referenced in a task but not present in the PRD, record it as a phantom reference (a task claims to address a requirement that does not exist).

### Step 7: Scope Check

Evaluate each task for size and clarity:

1. **Too large.** Flag tasks where the `estimated_complexity` is `large` AND the `files_affected` list contains more than 5 files, OR the `description` describes multiple distinct units of work that could be separated. Suggest how the task could be split.
2. **Too vague.** Flag tasks where:
   - The `description` is shorter than 50 characters.
   - The `acceptance_criteria` list contains only one entry that is not a testable condition (e.g. "Task is complete" or "Code works").
   - The `files_affected` list is empty or contains only generic paths (e.g. "src/" without a specific file name).
3. **Missing test tasks.** For each implementation task (a task that creates or modifies source code), check whether there is a corresponding test task that depends on it. If an implementation task has no associated test task, flag it as a warning.
4. **Role validity.** Verify every `suggested_role` value is one of: `python`, `nodejs`, `devops`, `frontend`, `general`, `docs`, `skills`. Flag any invalid roles.
5. **Complexity validity.** Verify every `estimated_complexity` value is one of: `small`, `medium`, `large`. Flag any invalid values.
6. **Field completeness.** Verify every task has all required fields (`id`, `title`, `description`, `requirements`, `dependencies`, `suggested_role`, `acceptance_criteria`, `estimated_complexity`, `files_affected`) and that list fields (`requirements`, `acceptance_criteria`, `files_affected`) each have at least one entry.

### Step 8: Determine Verdict

Based on the analysis from Steps 4-7, assign one of two verdicts:

**`PASS`** -- The task list is ready to proceed to execution. All of the following must be true:

- The dependency graph is a valid DAG with no cycles.
- All dependency references point to existing task IDs.
- No ordering issues were found (no task references files not yet created by its dependency chain).
- Every PRD requirement is covered by at least one task.
- No phantom requirement references exist.
- No tasks are flagged as too vague.
- All tasks have valid roles, complexity values, and complete fields.

Note: Warnings (orphan tasks, missing test tasks, tasks flagged as too large) do not block a `PASS` verdict, but they must be documented in the review.

**`REVISE`** -- The task list needs changes before proceeding. One or more of the following are true:

- The dependency graph contains cycles.
- Dependency references point to non-existent task IDs.
- Ordering issues were found (tasks reference files not created by their dependency chain).
- PRD requirements are not covered by any task.
- Phantom requirement references exist.
- Tasks are flagged as too vague.
- Tasks have invalid roles, complexity values, or incomplete fields.

When issuing a `REVISE` verdict, provide specific, actionable feedback for each issue. Each piece of feedback must state: the task ID (or requirement ID) affected, what the problem is, and exactly what needs to change. Do not give vague suggestions like "fix the dependencies" -- instead say "TASK-005 depends on TASK-003 and TASK-007, but TASK-003 creates `src/auth/handler.py` which TASK-005 also modifies; add TASK-003 to TASK-005's dependencies".

### Step 10: Report

Report a brief summary to the user:

- The verdict (`PASS` or `REVISE`)
- Dependency validation result (valid DAG or number of cycles / invalid references found)
- Ordering check result (clean or number of ordering issues found)
- Coverage statistics (X of Y PRD requirements covered)
- Scope check result (number of tasks flagged as too large, too vague, or missing test tasks)
- Number of action items (if any)
- Next step guidance:
  - If `PASS`: "Run `/jp-execute <phase-path>` to begin implementation."
  - If `REVISE`: "Update `tasks.yaml` to address the action items (or re-run `/jp-task-list <phase-path>` to regenerate), then re-run `/jp-task-review <phase-path>`."

## Guidelines

- **Validate structure, not design.** This skill checks whether the task list is well-formed, correctly ordered, and complete. It does not evaluate whether the tasks are the right approach to implementing the requirements -- that is a judgment call for the developer.
- **Be strict on blocking issues, lenient on warnings.** Cycles, invalid references, ordering problems, coverage gaps, vague tasks, and invalid fields are blocking issues that mandate a `REVISE` verdict. Orphan tasks, missing test tasks, and oversized tasks are warnings that should be documented but do not block a `PASS`.
- **Specificity over volume.** Every finding must include the affected task ID or requirement ID, a clear description of the problem, and a concrete suggestion for resolution.
- **REVISE feedback must be actionable.** Each action item must describe the exact change needed. "Fix TASK-005 dependencies" is not actionable. "Add TASK-003 to TASK-005's dependencies because TASK-005 modifies `src/auth/handler.py` which is created by TASK-003" is actionable.
- **Do not invent requirements.** If a task addresses work that is not in the PRD, flag it as a phantom reference. Do not add new requirements or suggest tasks beyond what the PRD specifies.
- **The review must stand alone.** A reader should be able to understand every finding in the review document without needing to cross-reference the task list or PRD.
